# HarmonyOS 发版前自动化执行手册

## 目标

在每次发版前，使用可重复执行的自动化脚本完成 P0 冒烟与核心回归，减少“改了但没及时发现”的问题。

## 技术栈结论（v1.1）

- **设备与安装基础层**：`hdc`
  - 负责设备连接、安装/卸载 HAP、启动应用、抓取 `hilog`。
- **UI 回归执行层**：`hmdriver2`（Python）
  - 负责页面交互、断言、截图、失败定位。
- **官方调度层（后续可选）**：OpenHarmony `developertest/xDevice`
  - 当前版本不强依赖；后续如需统一跨设备调度和更完整报告，再接入。

## 用例分层

- `tests/smoke/`：5-8 分钟可跑完，发版必跑，失败即阻断。
- `tests/e2e/`：主流程回归与关键缺陷回归，发版建议全跑。

## 当前发版门禁

- `P0` 冒烟全部通过才允许发版。
- 以下问题一票否决：
  - 主流程无法走通（开始、结束、保存结果）。
  - 配置保存后不生效（如默认时长不刷新）。
  - 保存成功提示与真实数据状态不一致。

## 执行入口

- 一键执行脚本：`scripts/release/pre_release_check.sh`
- 默认执行：
  1. 环境检查（Python + hdc + 设备在线）
  2. P0 冒烟
  3. E2E 回归

## 产物归档

- 统一归档到 `artifacts/automation/<timestamp>/`
  - `hilog.log`
  - 每条失败步骤截图
  - `summary.json`
- 命名建议包含版本号与设备 SN，便于追溯。

## 维护约定

- 每修复一个线上/审核问题，至少补一个自动化回归用例。
- 用例优先使用稳定文案或稳定标识定位，避免纯坐标点击。
- 脚本报错必须返回非零退出码，确保可被 CI/发版脚本拦截。
